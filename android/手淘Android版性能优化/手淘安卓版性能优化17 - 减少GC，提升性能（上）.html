手淘安卓版性能优化17 - 减少GC，提升性能（上）

    前面已经涉及到很多和性能相关的优化点，今天再来谈一下另外一个性能中的关键因素 - GC。之前也写过一篇文章，涉及到GC，《手淘安卓版性能优化6 - Dalvik虚拟机和ART运行时的介绍及对性能的影响》。在进行垃圾回收的过程中，或多或少的都会引起停顿，所以减少GC次数，尤其是减少GC_FOR_ALLOC类型的GC，可以显著提高系统的性能。GC的优化涉及到的面会很多，对Java等内存的分配需要有比较多的了解，对写代码的开发人员要求也会比较高。接下来就通过几个方面来谈下如何优化GC。

一、不可忽视的小对象分配

    下面我们先来看两个简单的循环用例。
     代码段1：

		long sum = 0L;
		for(long i = 0; i < Integer.MAX_VALUE; i++){
			sum += i;
		}

     代码段2：

		Long sum = 0L;
		for(long i = 0; i < Integer.MAX_VALUE; i++){
			sum += i;
		}

     上面两端代码只有一个字符的差别，long和Long的差别，但是两者的运行时间差别可不小，可谓失之毫厘，差之千里。在Nexus5 Android 5.0以上的设备上运行，代码1的运行时间是10110 ms，而代码2的运行时间是873821 ms，相差近87倍的时间。为什么一个字符的差别会导致那么大的差距呢？代码2中隐含了一次Java的包装类型自动装箱的操作，将long基本类型自动包装成Long类型，在每次循环中都会分配一个Long对象，额外多了一次对象的分配，当然这样短生命周期的对象频繁分配，自然会引起内存抖动，增加GC的负担。如下图所示，1秒时间大约执行了10次GC，频繁的GC也是导致性能的下降主要因素：


    从这里也可以看出，虽然很小的Long对象分配每次才16字节，但是快速频繁的分配依然会造成性能的快速下降。这里只是一个简单的Demo，在复杂的App中，分配的对象会更多，分配的字节数也会更大，对性能的影响是显而易见的。所以我们引出了另外一个话题。

二、减少对象的分配

   减少对象的分配，可以减轻GC的压力，在我们写代码的过程中，每次new对象的时候就应该下意识的去思考一下是不是必须要分配这个对象，如何来分配这个对象会最合适，养成代码细节上的习惯。例如在我们定义变量的时候或者定义函数返回值的时候，经常可以看到Boolean类型，但是实际上却是用boolean类型就可以了。这个时候的Boolean定义无疑是一种浪费。其他包装类型也会遇到这个问题。
    在开发过程中字符串的使用也是非常多的，“+”号自然是字符串拼接最人性化的做法，但是凡事有得有失，“+”号对字符串的拼接会被编译器自动优化，采用StringBuilder来实现，字符串常量的相加除外。当我们的代码中到处都是字符串处理的+号的时候，那性能也自然会受到很大的影响。
    在应用层的开发过程中，事件监听的的实现是很常见的，写法也有多种。有些时候我们会为几十个按钮创建几十个 new OnClickListener() ，这样的写法除了后续维护不容易找到对应代码外，还额外增加了不少内部匿名类的对象分配，增加了内存使用和代码量，也增加了方法数量。 从性能、内存、Dex文件大小、以及方法数上面考虑，都更应该使用在一个文件中只创建一个OnClickListener()的写法，然后通过View的id加以甄别。    
    在应用层开发过程中，还有一个是非常常用的，Handler 和 Message 。在异步处理上和定时操作的实现上都非常的方便。 这里先主要讨论的是以Runnable形式发送的消息，如Handler.post(Runnable r)、Handler.postAtTime等，另外一些不是通过Handler的写法就是View.post(Runnable r)及Activity.runOnUiThread(Runnable r)。 这些用法有时候可以在代码中带来方便，通过随时的创建一个匿名类来实现接口。但是我在很多中间件的代码中也发现了大量这样的写法，很多是在高频率的函数中。那么问题来了，高频率的函数中，每次调用都需要分配一个Runnable对象，这也是一种较为隐晦的多余对象分配，过犹而不及，反而给性能带来影响。绝大部分情况下都是可以通过Handler.sendEmptyMessage(int what)及Handler.sendMessage(Message msg)代替。因为Message中可以传递一些参数，int和boolean等少数类型的参数直接可以传递，而obj参数可以传递整个Object。即使有较多的参数需要传递，在高频率下也应该使用自定义的类并通过obj参数来实现对象的传递，并建立该对象的复用池。当然在sendMessage(Message msg)的时候也还是有一个地方需要注意的，Message对象也可以复用，而且系统已经为我们实现了该对象的复用池。所以我们在写代码的时候，不应该直接new Message() ，应该通过Message的obtain方法或者Handler的obtainMessage方法来复用Message，减少该对象的分配。

    另外，在主线程中，我们要尽量少的来分配对象，以减少GC引起的停顿，前面也已经多次提到过在GC过程中会对Heap加锁，请求分配的线程会被挂起等待。所以特别是在事件处理函数，绘制等高频率及要求快速响应的函数中，如onDraw、onTouchEvent等函数中，不应该分配对象，而应该在事先分配好对象。

    当然，代码中很多地方都需要注意对象的分配，这里只提到了几个典型的地方，后面在容器的使用过程中，也还会再涉及到对象的分配。

三、做好对象的复用

    在前面第二点已经提及了对象的复用，在循环体中以及在高频率函数中，如果频繁分配对象是件很糟糕的事情，我们在写代码的时候要注意这些细节点，比如在循环体内可以复用的对象，应该在循环体前面分配，例如下面演示了对Formatter对象和StringBuilder的复用：

		String str = "abcdefg %d %s";
		StringBuilder strBuilder = new StringBuilder(150);
		Formatter formatter = new Formatter(strBuilder);

		for (int i = 0; i < 100000; i++) {
			formatter.format(str2, i, "abc");
			strBuilder.append(floatStr);
			strBuilder.substring(0);
			strBuilder.setLength(0);
		}

    上面的代码涉及到了两个对象的复用，在循环的过程中并没有再分配StringBuilder对象和Formatter对象，虽然在format函数内部还会再分配一些内部的匿名类。在高频率的函数中，我们也应该使用各种手段来复用对象，减少不必要的对象分配。
    系统也有为我们做了对象的复用池：
    1) 线程池。
    2) Long、Integer、Short等包装类型内部有256个缓存对象；Boolean包装对象也有缓存，我们应该用包装对象的valueOf创建对象，而不是直接new。我们可以看下Long的源码，就一目了然了：

    /**
     * Returns a {@code Long} instance for the specified long value.
     * <p>
     * If it is not necessary to get a new {@code Long} instance, it is
     * recommended to use this method instead of the constructor, since it
     * maintains a cache of instances which may result in better performance.
     *
     * @param v
     *            the long value to store in the instance.
     * @return a {@code Long} instance containing {@code v}.
     * @since 1.5
     */
    public static Long valueOf(long v) {
        return  v >= 128 || v < -128 ? new Long(v) : SMALL_VALUES[((int) v) + 128];
    }

    /**
     * A cache of instances used by {@link Long#valueOf(long)} and auto-boxing.
     */
    private static final Long[] SMALL_VALUES = new Long[256];

    static {
        for (int i = -128; i < 128; i++) {
            SMALL_VALUES[i + 128] = new Long(i);
        }
    }

     3) 容器对象通过clear函数可以清空内容，再做复用。
     4) StringBuilder等也可以通过setLength(0)删除原内容做复用。
     5) 布局的复用，除了ListView等系统控件内部的复用外，如果自己有动态添加和删除的，也要做这方面的复用来提升性能。
     6) 图片内存的复用，可以通过BitmapFactory.Options.inBitmap参数实现。在Android4.4以下，系统要求相同尺寸的Bitmap可以复用，在Android4.4以上只要该图片数据块能放得下就可以复用了。另外Android5.0以下，图片内存还可以利用将Bitmap解码到Ashmem（Anonymous Shared Memory）来减少对内存的占用，从而减小GC的压力。其实Android的堆内存本身也是Ashmem，从Android 5.0开始，ART已经将大块的图片内存利用Large Object Space来存储，Large Object Space直接使用的就是Ashmem内存，所以在Android5.0以上，系统在图片等大内存的分配和回收上已经做了很大的性能提升。

三、尽早释放无用对象的引用

    在代码运行的过程中，对于较大的内存对象，我们可以手动将该对象设置为NULL，这样可以提前释放该对对象的引用，有利于及时的回收。
    这里还有涉及到另外一个概念会对GC产生影响 --- 栈帧（Stack Frame）。栈帧是用于支持虚拟机进行方法调用和方法执行的数据结构，存储了方法的局部变量表、操作数栈、动态连接和方法返回地址等信息。栈帧的局部变量表中的Slot是可重用的，方法体中定义的变量，其作用域并不一定会覆盖整个方法体，如果当前字节码PC计数器的值已经超出了某个变量的作用域，那么这个变量对应的Slot就可以交给其他变量使用。这样的设计不仅仅是为了节省栈空间，在某些情况下Slot的复用会直接影响到系统的垃圾收集行为。我们以几个例子来说明。

publicclass LocalVariableTableTest {

    publicstaticvoid main(String[]args) {

        byte[] testholder = newbyte[10 * 1024 * 1024];

        System.gc();

    }
}

    如上代码很简单，分配了10MB的数据，然后通知虚拟机进行垃圾回收。但是在System.gc()运行后并没有回收掉这10MB的内存。这里可以理解，在执行回收的时候，因为变量testholder还在作用域内。接着我们修改一下代码如下：

publicclass LocalVariableTableTest1 {

      publicstaticvoid main(String[]args) {

        {

            byte[] testholder = newbyte[10 * 1024 * 1024];

        }

        System.gc();

      }
}

    加入了花括号之后，testholder的作用域被限制在花括号之内，从逻辑上讲，在执行System.gc()的时候，testholder已经不可能再被访问了，但此时该内存还是没有被回收掉。我们再看第三种情况，代码如下：
publicstaticvoid main(String[]args) {

        {

               byte[] testholder = newbyte[10 * 1024 * 1024];

        }

        int a = 0;

        System.gc();

}

    在后面加了一个变量的分配，执行后就可以顺利的回收了。上面三段代码中，testholder能否被回收的根本原因就在于：局部变量表中的Slot是否还存有关于testholder对象的引用，而局部变量也是GC Roots的一部分。第二段代码虽然已经离开了testholder的作用域，但在此之后，没有任何对局部变量表的读写操作，testholder原本所占用的Slot还没有被其他变量所复用，所以导致这个时候还不能及时的进行回收。不过这种关联没有被及时打断的情况，在绝大部分情况下影响都是轻微的，毕竟在我们App中，线程后面还会有不停的函数执行着。
    主动对后续不会使用而又占据大块内存的变量及时设置为NULL，有时候还是必要的，比如在此之后执行了很耗时的代码逻辑，这样就可以加快内存的回收了。当然这只是某些大块内存被占用情况下的一个手段，并非是要作为一个普遍的编码方法来推广。一是因为赋NULL值的操作在经过虚拟机JIT编译器优化之后会被消除，当然只有达到一定条件的代码段才会被JIT优化，另外更主要的原因是频繁的赋值NULL也在多线程编程中埋下定时炸弹，有可能时不时的触发到NullPointerException异常，使用不当，反而对稳定性产生不利影响。

四、减少代码中的主动GC调用

    在有些代码中能够发现System.gc()或者Runtime.gc()的代码调用，这种主动通知垃圾回收的做法在Android环境下其实大可不必。Android（特别是Android 5.0以下）的垃圾回收策略还比较初级，性能并不高，额外触发GC造成更多的停顿。另外gc()函数的作用只是提醒虚拟机希望进行一次垃圾回收。但是它不能保证垃圾回收一定会进行，而且具体什么时候进行是取决于具体的虚拟机的，不同的虚拟机有不同的对策。所以我的建议是不需要人为的干预垃圾回收，交给虚拟机自己去管理。
